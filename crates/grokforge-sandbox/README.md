# grokforge-sandbox

`grokforge-sandbox` turns a platform-independent `SandboxPolicy` into command isolation. It reports the selected backend and whether enforcement is active; a missing backend is not treated as an enforced sandbox.

## Current backends

| Host | Backend | Current behavior |
|---|---|---|
| Linux | bubblewrap (`bwrap`) | Requires a validated system installation of non-setuid `bwrap` version 0.11.2 or newer that passes a namespace self-test. The host root is read-only, configured workspace roots are writable, protected metadata is rebound read-only, common host socket locations are hidden, explicit home/session credential paths are masked, all child capabilities are dropped, and the network namespace is isolated unless the policy explicitly grants full network access. |
| macOS | Seatbelt (`/usr/bin/sandbox-exec`) | Requires a successful startup self-test. Generated profiles confine writes, protect metadata, deny configured and explicit home/session secret-path reads, provide private temporary storage in workspace-write mode, deny Mach service lookup and Apple Event sends, and deny network access unless the policy grants it. This is a policy sandbox rather than a process namespace; process visibility and inherited channels remain separate boundaries. |
| Windows and other hosts | None | There is no native enforcing command backend. Policies that request confinement are refused by the passthrough runner. WSL2 can use the Linux backend when a compatible `bwrap` is installed and usable. |

Landlock and seccomp are design-roadmap work; they are not part of the current Linux implementation.

## Fail-closed behavior

The fallback runner reports `enforced = false`. It executes only a policy with no remaining confinement requirement at all. Normal read-only and workspace-write policies are rejected, as is a danger/full-access policy while protected paths remain present. GrokForge's `yolo` policy retains protected Git metadata, so its shell also fails closed without an enforcing backend. This can make shell execution unavailable on an unsupported or misconfigured host, but it does not silently discard a requested policy.

Every backend validates its public boundary before compiling or bypassing a policy: the command cwd and readable/writable roots must be absolute, existing directories, protected paths must be absolute, and unreadable globs must compile. Only an exact danger policy with `/` read/write roots, full network, and no protected or unreadable rules may run without an enforcing wrapper.

Every spawned shell and sandbox wrapper also starts with an empty environment. GrokForge restores only a validated absolute-directory `PATH`, narrowly validated standard locale and terminal variables, and a normalized `NO_COLOR` flag. Host `HOME`, `TMPDIR`, proxy settings, credential helpers, cloud/CI variables, dynamic-loader settings, and all other ambient variables are omitted. Enforcing backends inject only their own private temporary-directory value where applicable.

On Linux, a writable workspace cannot be protected by a bind mount if a protected path does not yet exist, and a path rule cannot safely cover aliases into a protected tree. GrokForge validates every existing protected tree, including external linked-worktree/common Git directories, on both Linux and macOS. Symlinks, hard-linked files, non-regular entries, scan errors, and trees larger than 100,000 entries fail closed. Linux additionally rejects a missing protected path below a writable root because a command could otherwise create it after mount setup. Consequently, Linux workspace-write shell commands are unavailable in a non-Git directory (or a nested workspace without its own `.git` entry) until an ordinary `.git` entry exists at the workspace root. External protected trees under `/tmp` or `/run` are staged before those host runtime trees are hidden and restored read-only inside the sandbox.

Before either enforcing backend runs a confined command, GrokForge scans the exposed workspace tree. Regular files (and symlink inodes) with more than one hard link are rejected because the same inode may be exposed or mutated through another name; this applies to both Bubblewrap mounts and Seatbelt path rules. Unix sockets, FIFOs, and device/special entries are rejected because read-only filesystem rules and a Linux network namespace do not isolate filesystem IPC or device access. Ordinary symlinks are resolved without walking unbounded external trees: broken links and links whose final target is a socket/FIFO/device fail closed, links to regular files remain usable, and directory links are accepted only when their physical target is inside an already scanned confined root. Read-only policies perform this preflight even when no unreadable globs are configured, deriving the project root from cwd and its lexical `.git` protection entry. Consequently, read-only commands also fail closed on repositories containing hardlinks, FIFOs, sockets, devices, unsafe symlinks, or more than 100,000 scanned entries. A filesystem-wide Bubblewrap retry scans only the active workspace because `/` cannot be walked within a useful bound.

An approved filesystem-wide retry remains a workspace-write policy with `/` as its writable root: bubblewrap makes the root bind writable for that command while retaining network isolation, protected metadata, socket hiding, and secret masks. If its cwd is below `/tmp` or `/run`, that cwd is staged and restored before command execution.

## Boundaries and limitations

- Bubblewrap cannot express secret globs directly. GrokForge performs a bounded, ASCII-case-insensitive workspace scan without descending through symlinked directories and masks currently existing matches (up to its scan and mask limits). Discovery checks both each lexical name and its resolved physical target. It also masks a bounded explicit set of existing credential locations derived from `HOME`, XDG directories, and common override variables: GrokForge session rollouts, SSH, AWS, GnuPG, Docker/container, Kubernetes, cloud CLI, package-manager, Git credential, password-store, and desktop-keyring paths. Directories become empty read-only mounts and files are rebound to `/dev/null`. Exact absolute unreadable rules for GrokForge's external session/worktree stores remain active in danger mode even though unrelated home credentials are intentionally exposed there.
- Seatbelt translates configured secret globs into safely escaped, ASCII-case-insensitive profile regular expressions that deny both reads and writes, preventing a matched name from being renamed or replaced before reading. The same bounded discovery resolves every existing match into an explicit physical and lexical read/write denial, so a symlinked secret target cannot be read through its ordinary name. Ancestor directory entries between protected/private targets and their broadest writable root are frozen with literal write denials (up to 2,048 parameters), preventing a parent rename from relabeling the protected target while leaving ordinary descendant and sibling writes available. Hard-linked regular files in scanned roots and external directory symlinks fail closed even in read-only mode. A workspace nested inside a directory that must be hidden is rejected on both enforcing backends rather than producing a mount/profile conflict.
- The explicit credential list is intentionally bounded, not a claim to discover every application-specific secret. Every selected credential/session directory is recursively validated before its parent mask or denial is installed. Nested symlinks, hard-linked entries, inspection errors, and trees larger than 100,000 entries fail closed, including arbitrarily named nested files. A single-name socket/FIFO/device below a retained parent directory is safe because that parent is hidden; if such a special entry has another hard-link name, validation rejects it. Missing paths and files created elsewhere after sandbox setup are not masked, pre-spawn checks cannot stop a separate same-user process racing a path replacement, and project glob scanning does not traverse the entire host root. The context redactor remains a separate best-effort layer.
- Seatbelt does not create PID, IPC, or Mach namespaces. Every wrapped profile denies new Mach service lookup, Apple Event sends, and signals to processes outside the same sandbox while allowing child/process-group signaling within it. Process visibility and inherited file descriptors or Mach rights remain separate boundaries.
- `DangerFullAccess` permits filesystem access outside the workspace and full network access and disables the default unreadable-secret globs, but GrokForge retains protected metadata as deny-write. Because that invariant still requires the wrapper, Linux also keeps host `/tmp` and `/run` hidden behind private mounts; yolo therefore does not expose host temp files or Docker/DBus/runtime sockets. It still needs an enforcing backend. Only an internal policy with no protected paths, no unreadable globs, and full network takes the truly unwrapped fast path.
- The sandbox covers spawned commands. Host-side file tools have their own path checks, and explicitly trusted external processes such as MCP servers are separate trust boundaries.
- Proxy-routed/domain-allowlisted networking is not implemented. A non-full network policy is isolated instead of being silently broadened.

## Host Git support

Git mutations are outside this crate's command sandbox and run through GrokForge's separate trusted-host Git boundary. That boundary currently accepts validated system Git executables on Unix only. Native Windows therefore has no auto-commit, `/undo`, or subagent worktree support; use WSL2 for the Linux implementation.
